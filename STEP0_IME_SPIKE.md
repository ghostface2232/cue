# 단계 0 — IME 조합 신호 스파이크: 실행 & 관찰 프로토콜

> **이것은 버리는 코드다.** `Pages/ImeSpikePage.xaml(.cs)`, `MainWindow.xaml`의 `IME 스파이크`
> 네비 항목, `MainWindow.xaml.cs`의 `imespike` 분기, 그리고 이 문서는 단계 0 판정이 끝나면
> 통째로 삭제한다. 위에 어떤 실제 기능도 쌓지 않는다.

## 왜 하는가

이 계획 전체의 트리거 설계는 세 가정 위에 선다. 코드를 더 짓기 전에 **Windows에서 실제 한글
IME로 직접 관찰**해 못 박는다.

1. **신호 존재**: `TextCompositionStarted/Changed/Ended`가 한글 IME에서 실제로 발화하고,
   "지금 조합 중인가"를 깨끗한 창으로 잡을 수 있는가?
2. **재틴트 안전**: commit/공백/멈춤 트리거에 전체 재틴트할 때 **caret이 튀는가**, **Ctrl+Z
   (native undo)** 스택이 깨지는가?
3. **leak 없음**: 조합 중 글자(`ㄱ→가→갈`)에 서식이 새는가?

## 실행

Windows(개발자 모드)에서 저장소 루트에서:

```powershell
.\BuildAndRun.ps1 Cue.csproj
```

앱이 뜨면 좌측 네비 하단 **"IME 스파이크"** 를 연다. 입력기를 **한글**로 바꾸고(한/영 키)
상단 RichEditBox에 타이핑하면서 **하단 로그**와 **상태 줄**(`IsComposing` / `caret` / `len` /
`compositions` / `lastTrigger`)을 본다. 디버거는 필요 없다 — 모든 신호가 화면에 찍힌다.

화면 토글:
- **비조합일 때만 재틴트**: 기본 ON. **끄면 조합 중에도 칠해 leak을 직접 관찰**한다.
- **틴트를 undo 그룹으로 격리**: ON/OFF를 바꿔 Ctrl+Z 거동을 비교한다.

## 관찰 시나리오 (순서대로)

### A. 조합 신호가 발화하는가 (가정 1)
1. 토글 둘 다 기본값(게이트 ON, undo 격리 ON).
2. `내일 오후 3시 회의` 를 천천히 친다.
3. **본다**: 글자마다 `CompositionStarted` → `CompositionChanged…` → `CompositionEnded`가
   로그에 찍히는가? 상태 줄 `IsComposing`이 조합 중 `true`, 글자 확정 후 `false`로 깔끔히
   토글되는가? `compositions` 카운트가 친 글자 수만큼 늘어나는가?
   - ✅ 찍힌다 → 신호 확보. 가정 1 통과.
   - ❌ 안 찍힌다 / `IsComposing`이 안 바뀐다 → 신호 없음. **대안 경로**(아래) 필요.

### B. caret이 튀는가 (가정 2-a)
1. `내일 오후 3시 회의` 입력 후, **문장 중간**(예: `오후`와 `3시` 사이)을 클릭해 caret을 둔다.
2. 끝에 공백을 하나 치거나 500ms 기다려 재틴트를 유발한다.
3. **본다**: 로그의 `tint […] caret a,b -> c,d` 줄. `-> ` 뒤 값이 앞 값과 같은가?
   `<<< CARET MOVED`가 뜨면 caret이 튄 것이다.
   - ✅ 안 튐 → 통과.
   - ⚠️ 튐 → 재틴트 전후 caret 저장·복원으로 잡히는지, 아니면 부분 재서식이 필요한지 기록.

### C. Ctrl+Z(native undo)가 깨지는가 (가정 2-b)
1. **undo 격리 ON** 상태로 `금요일 회의` 입력 → 재틴트가 돌게 둔다 → `Ctrl+Z`를 누른다.
2. **본다**: Ctrl+Z가 **내가 친 텍스트**를 되돌리는가(정상), 아니면 **틴트(색)** 만
   되돌리거나 아무 일도 안 일어나는가(오염)?
3. **undo 격리 OFF** 로 바꿔 같은 입력 반복, 거동을 비교한다.
   - ✅ 텍스트가 정상 되돌려짐 → 통과(어느 토글에서 정상인지 기록).
   - ⚠️ 색만 되돌려지거나 텍스트 undo가 망가짐 → 어느 토글에서 그런지 기록.

### D. 조합 글자에 서식이 새는가 (가정 3)
1. **게이트 ON**(기본)으로 `갈비` 처럼 조합이 여러 단계인 글자(`ㄱ→가→갈→갈비`)를 친다.
   조합 중에는 색이 안 들어가고, 확정(commit) 후에만 색이 정리되는가? → 기대 동작.
2. **게이트 OFF**(leak probe)로 같은 입력을 친다. 조합 중인 반쪽 글자에 액센트가 번지거나
   IME 조합이 깨지는가?
   - ✅ 게이트 ON에서 leak 없음 → 통과(게이트 설계가 유효).
   - ⚠️ 게이트 ON인데도 leak/깨짐 → 트리거를 commit 단위로만 더 좁혀야 함을 기록.

### E. `금요일 → 금요일마다` 전이(부수 확인)
1. `금요일` 입력 → 확정 후 색을 본다(throwaway 매처라 `금요일`/`요일`이 칠해짐) →
   이어서 `마다` 를 붙인다 → 재틴트 후 색이 갱신되는가?
   - 이 스파이크의 매처는 dumb하지만, **재틴트가 수정 지점에 매끄럽게 반응하는지**의
     감만 본다(정확한 kind 구분은 단계 1·3에서).

## 판정표 (채워서 회신)

| # | 가정 | 결과 | 메모 (어느 토글/입력에서) |
|---|------|------|--------------------------|

## 판정 → 분기 (이 결과로 단계 1·2·3 경로 확정)

- **전부 ✅** → 단계 3의 **"full reset + re-tint" 단일 진실 경로**로 진행.
- **2a/2b ⚠️ (caret/undo 훼손)** → 단계 3을 **부분 재서식**(바뀐 토큰 span만 갱신)으로 설계.
- **3 ⚠️ (게이트로도 leak)** → 단계 2 트리거를 **IME commit 단위로만** 더 좁힘.
- **1 ❌ (신호 없음)** → 대안 신호 조사: `CoreTextEditContext` 직접 사용, 또는 조합 중
  중간 글자(자모 단위 변화)를 `TextChanged`로 식별. 확보 후에야 단계 2 트리거 재설계.

회신해 주시면 결과를 이 문서 하단에 박제하고, 스파이크 코드를 제거한 뒤 그 판정에 맞춰
단계 1(파서 계약 + 엔진 리팩터)로 들어가겠습니다.

## 환경 메모

이 스파이크는 Windows에서만 빌드·실행된다(WinUI 3 / Windows App SDK, `net10.0-windows10.0.26100`,
개발자 모드 필요). 작성·커밋은 Linux 컨테이너에서 했으므로 **컴파일 검증은 하지 못했다** —
빌드가 막히면 오류 메시지를 회신해 주면 바로 고친다.
